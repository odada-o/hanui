## 1. 경쟁 환경 개요

### 1.1 시장 상황

```
KRDS(대한민국 정부 표준 디자인 시스템) 시장:
- 2023년 공식 발표
- 모든 공공 웹사이트 적용 의무
- 시장 규모: 연 9,000억원 (웹/앱 프로젝트)

현재 상황:
✓ 공식 가이드: HTML만 존재
✗ React/Vue 구현체: 커뮤니티 주도
✗ 통합 플랫폼: 없음
✗ 생태계: 미비

→ 블루오션 시장
```

### 1.2 경쟁자 분류

```
직접 경쟁자 (KRDS 라이브러리):
├─ KRDS-community/krds-react (Star ~50)
├─ @krds.ui/vue (Vue 3)
└─ @gracefullight/krds (MUI 기반)

간접 경쟁자 (글로벌 UI 라이브러리):
├─ PrimeVue/PrimeReact (상업용)
├─ Chakra UI (오픈소스)
├─ shadcn/ui (오픈소스)
└─ Material-UI (오픈소스)

대체재:
└─ 직접 구현 (가장 큰 경쟁자!)
```

## 2. 직접 경쟁자 분석

### 2.1 KRDS-community/krds-react

#### 기본 정보

```
GitHub: https://github.com/KRDS-community/krds-react
Stars: ~50 (2024년 1월 기준)
마지막 업데이트: 활발
라이선스: MIT
NPM: @krds-community/react
```

#### 강점 ✅

```
✓ 커뮤니티 주도 (여러 개발자 참여)
✓ KRDS 가이드 준수
✓ TypeScript 지원
✓ Storybook 문서
✓ 기본 컴포넌트 20개+
```

#### 약점 ❌

```
✗ 프로젝트 템플릿 없음
  → 개발자가 직접 세팅

✗ 문서가 기술 문서 수준
  → 기획자/디자이너가 보기 어려움

✗ 접근성 자동 검사 없음
  → 수동 확인 필요

✗ 생태계 없음
  → 라이브러리만 제공

✗ 유지보수 불확실
  → 커뮤니티 기여 의존
```

#### 위협도

```
★★☆☆☆ (중하)

이유:
- 기능은 있으나 생태계 없음
- 사용자 경험(UX) 부족
- 인지도 낮음 (Star 50)

→ 품질로 충분히 역전 가능
```

### 2.2 @krds.ui/vue

#### 기본 정보

```
NPM: @krds.ui/vue
Framework: Vue 3 + TypeScript
공식 주장: "대한민국 정부 표준 디자인 시스템 공식 컴포넌트"
문서: https://krds.initializer.org
```

#### 강점 ✅

```
✓ Vue 3 생태계 타겟
✓ TypeScript 완벽 지원
✓ 30+ 컴포넌트
✓ "공식" 포지셔닝 (마케팅)
```

#### 약점 ❌

```
✗ 실제로 정부 공식 아님
  → 커뮤니티 프로젝트

✗ 유지보수 의문
  → 30일 전 마지막 업데이트

✗ 사용 사례 부족
  → 레퍼런스 없음

✗ GitHub 비공개?
  → 오픈소스 아닐 가능성
```

#### 위협도

```
★★☆☆☆ (중하)

이유:
- Vue 시장은 React보다 작음
- 실제 "공식"이 아님
- 유지보수 불확실

→ React 시장에서 경쟁 없음
```

### 2.3 @gracefullight/krds

#### 기본 정보

```
GitHub: https://github.com/gracefullight/krds
Stars: ~30
기술: React + Material-UI
개발자: 개인 (gracefullight)
```

#### 강점 ✅

```
✓ MUI 기반으로 완성도 높음
✓ 개발자 1인이지만 꾸준한 업데이트
✓ TypeScript 완벽
✓ 실무 사용 경험 반영
```

#### 약점 ❌

```
✗ MUI 의존성
  → 번들 사이즈 큼 (~400KB)
  → MUI 라이선스 제약

✗ 커스터마이징 제한적
  → MUI 테마에 종속

✗ 개인 프로젝트
  → 장기 유지보수 불확실

✗ 생태계 없음
  → 라이브러리만 제공
```

#### 위협도

```
★★★☆☆ (중)

이유:
- 완성도는 높으나 MUI 의존성 부담
- 개인 프로젝트 한계
- 번들 사이즈로 공공 프로젝트 부적합

→ 경량화 + 생태계로 차별화 가능
```

## 3. 간접 경쟁자 분석

### 3.1 PrimeVue/PrimeReact

#### 강점

```
✓ 프리미엄 디자인
✓ 템플릿 마켓플레이스 (유료)

```

```
→ 템플릿 마켓플레이스 모델

→ 문서 사이트 UX
```

## 4. SWOT 분석

### Strengths (강점)

```
✅ KRDS 전문화
   - 유일한 KRDS 플랫폼
   - 공공 프로젝트 특화

✅ 한국 시장 이해
   - 접근성 기준 숙지
   - 공공 SI 프로세스 이해
   - 한국어 문서/지원

✅ 통합 솔루션
   - 라이브러리 + 도구 + 생태계
   - 원스톱 플랫폼

✅ 현대적 기술 스택
   - TypeScript, Tailwind, Next.js
   - 경쟁자 대비 가벼움
```

### Weaknesses (약점)

```
⚠️ 후발주자
   - 기존 라이브러리 이미 존재
   - 인지도 제로에서 시작

⚠️ 리소스 부족
   - 1인 개발 (초기)
   - 개발 속도 제한

⚠️ 검증 부족
   - 실사용 사례 없음
   - 신뢰도 구축 필요

⚠️ 수익 모델 미검증
   - Freemium 성공 여부 불확실
```

### Opportunities (기회)

```
🎯 공공 SI 시장 성장
   - 디지털 전환 가속화
   - KRDS 의무 확대

🎯 접근성 인증 강화
   - 법적 요구사항 증가
   - 자동화 도구 수요 ↑

🎯 AI 혁명
   - 코드 생성 트렌드
   - 생산성 도구 각광

🎯 오픈소스 문화 확산
   - 커뮤니티 기여 증가
   - 정부의 오픈소스 장려
```

### Threats (위협)

```
⚡ 경쟁자 모방
   - 기능 따라하기 쉬움
   - 차별화 유지 어려움

⚡ KRDS 가이드 변경
   - 대응 비용 발생
   - 호환성 깨질 수 있음

⚡ 시장 규모 과대 평가
   - 실제 지불 의사 낮을 수 있음
   - 무료 사용만 늘어날 위험

⚡ 공식 라이브러리 등장
   - 정부가 공식 제작 시
   - 시장 잠식 위험
```

## 5. 승리 전략

### 5.1 단기 전략 (1-6개월)

#### ① 품질로 압도

```
목표: "KRDS 하면 이거" 브랜드 구축

실행:
✓ 경쟁자 대비 10배 나은 문서
  - PrimeVue 수준 디자인
  - 인터랙티브 데모
  - 한영 지원

✓ 템플릿 제공 (유일)
  - create-krds-app
  - 즉시 사용 가능한 페이지

✓ 접근성 자동화 (유일)
  - VSCode Extension
  - 경쟁 우위 확보

✓ 1개월 안에 MVP 배포
✓ 매주 업데이트
✓ 피드백 즉시 반영

목표: 팬 확보

✓ Discord 활발히 운영

## 7. 리스크 대응

### 리스크 1: "경쟁자가 똑같이 만들면?"

**대응**:

```
A. 생태계는 복제 불가
   - Marketplace 네트워크 효과
   - 커뮤니티 충성도

B. 브랜드 선점
   - "KRDS Platform" 이름
   - "원조" 포지셔닝

C. 빠른 실행
   - 1년 앞서 나가기
   - 지속적 혁신
```

### 리스크 2: "공식 라이브러리 나오면?"

**대응**:

```
A. 협력 제안
   - "우리가 공식 되겠습니다"
   - 오픈소스로 기부

B. 차별화 유지
   - 공식보다 빠른 업데이트
   - 더 좋은 개발자 경험

C. 생태계 강점
   - Marketplace는 우리만
   - AI Generator는 우리만
```

### 리스크 3: "시장이 생각보다 작으면?"

**대응**:

```
A. 피봇 준비
   - 일반 디자인 시스템 도구로 확장
   - 글로벌 시장 진출

B. 비용 최소화
   - 오픈소스 + Freemium
   - 초기 투자 최소화

C. 다각화
   - 교육/컨설팅 수익
   - 엔터프라이즈 라이선스
