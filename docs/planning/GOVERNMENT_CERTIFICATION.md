## 2. 현실적인 접근 방법

### Level 1: 준공식 지위 확보 ⭐ (1년 목표)

#### 목표

```
"KRDS 공식은 아니지만, 정부가 인정하는 도구"

예시 포지셔닝:
- "행안부 KRDS 가이드 준수 검증 도구"
- "공공 웹사이트 접근성 자동 검사 서비스"
- "나라장터 입찰 시 활용 가능한 KRDS 플랫폼"
```

### 전략 1: 행안부 담당 부서 접촉

#### 접촉 대상

```
행정안전부:
- 디지털정부국
- 정부혁신조직실
- 전자정부 표준화 담당 부서

담당 업무:
- KRDS 관리
- 공공 웹사이트 표준화
- 전자정부 서비스 품질 관리
```

#### 접근 방식

**Step 1: 공문 발송**

```
제목: "KRDS 준수 자동 검증 도구 개발 협조 요청"

내용 구성:
1. 오픈소스 프로젝트 소개
   - HANUI 개요
   - 개발 목적 (공공 웹사이트 품질 향상)
   - 현재 진행 상황

2. KRDS 준수 여부 확인 요청
   - 가이드라인 준수 검증
   - 공무원 실사용 테스트 요청
   - 피드백 수렴

3. 협업 제안
   - 파일럿 프로젝트 협력
   - 정기 의견 교환
   - 공공 프로젝트 적용 사례 공유

첨부 자료:
- HANUI 소개서 (PDF)
- 기술 문서
- 데모 사이트 링크
```

**Step 2: 실무진 미팅**

```
준비물:
□ 데모 시연 (15분)
  - create-hanui-app 실행
  - 컴포넌트 쇼케이스
  - KRDS 준수 검증 시연
  - 접근성 자동 검사 시연

□ 발표 자료 (PPT)
  - 문제 정의 (현재 공공 웹사이트 개발의 어려움)
  - HANUI 솔루션
  - 차별화 포인트
  - 향후 계획

□ 레퍼런스 (있다면)
  - 파일럿 프로젝트 사례
  - SI 업체 피드백
  - 개발자 커뮤니티 반응

요청 사항:
- KRDS 가이드 준수 여부 확인
- 개선 사항 피드백
- 공공 프로젝트 적용 가능성 검토
```

**Step 3: 피드백 반영**

```
행안부 피드백 수렴:
- 가이드라인 해석 확인
- 미준수 항목 개선
- 추가 요구사항 반영

정기 보고:
- 월 1회 개발 진행 상황 공유
- 분기 1회 실무진 미팅
- 주요 마일스톤 달성 시 보고
```

### 전략 2: 파일럿 프로젝트 진행

#### 목표

```
"작은 성공 사례부터 축적"
→ 실제 사용 검증
→ 레퍼런스 확보
→ 신뢰도 구축
```

#### 대상 선정

**우선순위 1: 소규모 지자체**

```
타겟:
- 시군구 단위 (예: 성남시, 부천시, 수원시)
- 웹사이트 리뉴얼 예정인 곳
- 예산 제약이 있는 곳
- 담당자가 혁신적이고 오픈소스에 우호적인 곳

찾는 방법:
1. 나라장터에서 "웹사이트 구축" 입찰 공고 확인
2. 지자체 정보화 담당 부서 연락
3. 개발자 커뮤니티에서 공무원 개발자 접촉
4. IT 컨퍼런스에서 네트워킹
```

**우선순위 2: 중소 공공기관**

```
타겟:
- 교육기관 (평생교육원, 도서관)
- 문화기관 (박물관, 미술관)
- 복지기관 (복지관, 보건소)

특징:
- 규모가 작아 의사결정 빠름
- 혁신적 시도에 유연
- 예산 부담이 커서 무료 지원 매력적
```

#### 제안 방법

**무료 지원 제안서**

```markdown
제목: "HANUI 기반 웹사이트 구축 무료 지원 제안"

1. 제안 배경
   - HANUI 오픈소스 프로젝트 소개
   - 공공 웹사이트 품질 향상 목적
   - 실제 사용 사례 확보 필요

2. 지원 내용
   ✅ 웹사이트 개발 (무료)
   - HANUI 기반 구축
   - KRDS 가이드 100% 준수
   - 접근성 AA 등급 획득 보장
   - 반응형 디자인
   - 관리자 페이지 포함

   ✅ 기술 지원 (6개월)
   - 버그 수정
   - 기능 개선
   - 유지보수 가이드 제공

3. 요청 사항
   - 사례 연구 작성 허가
   - 레퍼런스 사용 허가
   - 피드백 제공
   - 접근성 인증 과정 공유

4. 예상 효과
   - 개발 비용 절감 (예상 ₩5,000,000)
   - 개발 기간 단축 (4주 → 2주)
   - 접근성 인증 통과율 95%+
   - 유지보수 용이

5. 일정
   - 착수: 즉시 가능
   - 1차 개발: 2주
   - 접근성 검수: 1주
   - 최종 납품: 4주 이내
```

#### 성공 사례 활용

**사례 연구 작성**

```markdown
제목: "○○시청 웹사이트, HANUI로 4주 만에 구축"

요약:

- 프로젝트: ○○시청 홈페이지 리뉴얼
- 기간: 4주 (기존 대비 50% 단축)
- 비용: ₩0 (기존 대비 100% 절감)
- 결과: 접근성 AA 등급 획득, 사용자 만족도 95%

Before (기존 방식):

- 개발 기간: 8주
- 비용: ₩10,000,000
- 접근성 검수: 지적사항 30개
- 수정 기간: 2주

After (HANUI 사용):

- 개발 기간: 4주
- 비용: ₩0 (무료 지원)
- 접근성 검수: 지적사항 2개 (경미)
- 수정 기간: 1일

담당자 인터뷰:
"HANUI 덕분에 예산을 절감하고 빠르게 웹사이트를
구축할 수 있었습니다. 특히 접근성 자동 검사 기능이
큰 도움이 되었습니다."

기술적 성과:

- KRDS 가이드 100% 준수
- 접근성 AA 등급 (행안부 인증)
- 성능 점수 95+ (Lighthouse)
- 반응형 디자인 (모바일 최적화)
```

**레퍼런스 홍보**

```
활용처:
1. HANUI 웹사이트에 사례 게시
2. 행안부에 성공 사례 보고
3. 다른 지자체에 마케팅 자료로 활용
4. IT 컨퍼런스에서 발표
5. 블로그/언론 기고

효과:
→ "○○시에서 사용한 HANUI"
→ 다른 공공기관의 신뢰 확보
→ 자연스러운 확산
```

### 전략 3: SI 업체 파트너십

#### 목표

```
"SI 업체를 통한 간접 확산"
→ 입찰 경쟁력 강화 도구
→ 자연스러운 시장 침투
```

#### 타겟 선정

**중소 SI 업체**

```
특징:
- 대기업: 자체 솔루션 있음 (경쟁 관계)
- 중소업체: 솔루션 부족, 협력 가능성 높음

찾는 방법:
1. 공공 프로젝트 많이 하는 업체 (나라장터 검색)
2. React/Vue 개발 업체
3. 프론트엔드 인력 부족한 업체
4. 혁신적인 중소 SI 업체

타겟 리스트 (예시):
- 지역 기반 SI 업체
- 스타트업 SI 업체
- 프론트엔드 전문 업체
```

#### 제안 내용

**Win-Win 파트너십**

```
SI 업체 제공 사항:
✅ HANUI 무료 교육 (2일 과정)
✅ 기술 지원 (Slack/Discord)
✅ 프로젝트 컨설팅
✅ 공동 마케팅 (로고 사용)
✅ 우선 지원 (Pro 플랜 무료)

SI 업체 이점:
1. 개발 속도 향상 (2배)
   - 프로젝트 세팅 시간 단축
   - 재사용 가능한 컴포넌트
   - 접근성 자동 검증

2. 입찰 경쟁력 강화
   - 제안서에 "HANUI 사용" 명시
   - 품질 보장 근거
   - 개발 일정 단축 제안 가능

3. 비용 절감
   - 개발 인력 절약
   - 접근성 검수 비용 감소
   - 유지보수 용이

4. 리스크 감소
   - 검증된 라이브러리
   - 커뮤니티 지원
   - 오픈소스 (락인 위험 없음)

요청 사항:
- 실제 프로젝트에 사용
- 피드백 제공
- 성공 사례 공유 (선택)
```

#### 효과

**시장 확산 메커니즘**

```
SI 업체 A: "HANUI로 빠르게 개발했어요"
         ↓
발주처:   "품질 좋네요, 다음에도 사용하세요"
         ↓
입찰 공고: "HANUI 또는 동등 이상의 라이브러리 사용"
         ↓
다른 SI 업체: "HANUI를 써야겠네"
         ↓
자연스러운 업계 표준화

결과:
→ 공공기관이 HANUI를 선호하게 됨
→ SI 업체들이 HANUI를 필수로 배움
→ 사실상 표준(de facto)으로 자리잡음
```

## 3. Level 2: 준공식 인정 (1-2년 목표)

### 전략 4: 공공 기관 등록

#### 공공데이터 포털 등록

```
사이트: data.go.kr

등록 대상:
- HANUI Validator API
- KRDS 준수 검증 API
- 접근성 자동 검사 서비스

효과:
✅ "공공 API"로 인정
✅ 공공 프로젝트에서 활용 가능
✅ 정부 플랫폼에 공식 노출
✅ 신뢰도 대폭 향상

절차:
1. 공공데이터 포털 가입
2. API 문서 작성
3. 등록 신청
4. 심사 (1-2개월)
5. 승인 후 공개
```

#### 한국지능정보사회진흥원(NIA) 협력

```
NIA 역할:
- 전자정부 표준 담당
- 오픈소스 프로젝트 지원
- 공공 SW 품질 관리

협력 방법:

1. 오픈소스 지원 사업 신청
   프로그램: "공공 SW 오픈소스 활성화"
   지원 내용: 개발 비용, 컨설팅, 마케팅
   신청 기간: 매년 상반기

2. 전자정부 표준 연계 과제
   프로그램: "전자정부 표준 고도화"
   제안: "KRDS 준수 자동 검증 표준"

3. 공공 SW 혁신 아이디어 공모
   주제: "AI 시대 공공 웹사이트 품질 관리"
   내용: HANUI + AI 협업 시연

효과:
✅ NIA 인증 = 준공식 지위
✅ 정부 예산 지원 (수천만원~수억원)
✅ 언론 홍보 효과
✅ 공공기관 네트워크 확보
```

```
효과:
✅ "TTA 인증 KRDS 플랫폼"
✅ 법적 근거 확보
✅ 공공 입찰 시 우대 가능
✅ 국제 표준으로 발전 가능
```

## 4. Level 3: 완전 공식화 (2-3년 목표)

### 전략 6: 오픈소스 기부

#### 개념

```
"HANUI를 정부에 기증하여 공식 프로젝트로"

예시:
- Apache 재단 방식
- Linux Foundation 방식
- 정부 주도 오픈소스
```

#### 제안 방법

```
제안서:
제목: "HANUI 프로젝트 행안부 기증 제안"

내용:
1. 기증 배경
   - KRDS 생태계 활성화 목적
   - 공공 웹사이트 품질 향상 기여
   - 오픈소스 정신 실천

2. 기증 내용
   - HANUI 전체 코드베이스
   - 지적재산권 이전
   - 문서 및 가이드
   - 커뮤니티 자산

3. 조건
   ✅ 오픈소스 라이선스 유지 (MIT)
   ✅ 커뮤니티 참여 보장
   ✅ 핵심 개발자로 계속 참여
   ✅ 상용 서비스 별도 운영 허용
      (Marketplace, Pro 플랜)

4. 운영 방안
   - 행안부: 공식 관리 주체
   - 원개발자: 기술 자문 및 개발
   - 커뮤니티: 자유로운 기여
   - 거버넌스: 오픈소스 위원회 구성

효과:
```

## 8. 핵심 메시지

### "공식 인증은 결과이지 목표가 아니다"

```
잘못된 접근:
"공식 인증을 받자" (목표)
→ 정치적, 오래 걸림, 불확실

올바른 접근:
"실제 가치를 만들자" (과정)
→ 시장 증명, 빠름, 확실
→ 인정은 자연스럽게 따라옴

예시:
- React는 Facebook 공식이 아님
- 하지만 사실상 표준
- 공식보다 중요한 것은 "실제 사용"

HANUI도:
- 행안부 공식을 너무 집착하지 않기
- "공공기관이 다 쓰는" 것이 목표
- 사실상 표준(de facto)이 되면 성공
```
