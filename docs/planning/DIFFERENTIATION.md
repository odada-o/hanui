## 1. 현황 분석

### 1.1 기존 KRDS 라이브러리의 공통 약점

```
모두 "컴포넌트 라이브러리"에만 집중

❌ 프로젝트 시작을 도와주지 않음
   → 개발자가 직접 세팅 (Webpack, Vite, ESLint...)

❌ 실무 템플릿 부족
   → "Button 컴포넌트는 있는데, 로그인 페이지는?"

❌ 접근성 검증을 개발자에게 맡김
   → 수동 체크 → 시간 낭비 → 검수에서 지적

❌ 문서가 딱딱함
   → 기술 문서 수준
   → "이거 써서 뭘 만들 수 있는지" 감이 안 옴

❌ 생태계 없음
   → 일회성 라이브러리
   → 커뮤니티 활동 미미
```

### 1.2 우리의 기회

```
✅ 시장 검증 완료
   - 여러 개발자가 각자 만듦 = 수요 있음
   - 하지만 "표준"은 없음

✅ 진입장벽 낮음
   - Star 50개 수준 = 아직 메이저 없음
   - 품질로 역전 가능

✅ 공공 SI 시장 거대
   - 연 9,000억원
   - KRDS 의무 사용
   - 모든 공공 프로젝트가 잠재 고객
```

## 2. AI 시대에도 유효한 근본적 경쟁력

### 2.0 왜 AI가 HANUI를 대체할 수 없는가?

#### 핵심 명제:

```
"AI는 코드를 잘 만들지만, 로컬 규범을 이해하지 못한다"
"AI 시대의 경쟁력은 생산성이 아니라, 문맥화된 표준을 정의하는 것"
```

---

#### ① "AI가 몰라도 되는 한국 공공 디자인 문맥"

**문제:**

```
ChatGPT, Cursor, v0.dev 등 AI 도구들:
✓ React 컴포넌트는 잘 만듦
✓ 글로벌 디자인 시스템(MUI, Chakra) 잘 알고 있음

하지만:
❌ 공공기관 웹 접근성 가이드 (행안부 기준)
❌ KRDS 표준 (문서형 접근성, 명도 대비)
❌ 행정용 색상 체계
❌ 한국형 웹호환성 기준
→ 학습 데이터가 거의 없음
```

**HANUI의 해답:**

```
HANUI = AI가 학습할 로컬 표준을 사람이 먼저 정리한 데이터셋

예시:
AI: "버튼 만들어줘"
→ 글로벌 기준의 버튼 (MUI 스타일)

HANUI 기반 AI: "KRDS 기준 버튼 만들어줘"
→ 명도 대비 4.5:1 준수
→ 행안부 접근성 가이드 준수
→ 문서형 접근성 마크업 포함
→ 한국 공공기관 UI 패턴 반영

→ 실제 현장에서 검수 통과 가능한 코드
```

**결론:**

> HANUI는 **"AI 이후 시대의 로컬 UI 표준화 리더"**가 될 수 있음

---

#### ② "공공사업 납품용 디자인 시스템"

**AI가 못하는 것:**

```
나라장터 입찰·공공 발주에서 요구되는 것:

필수 요건:
□ 명확한 출처 (디자인 시스템 버전 명시)
□ 공공 접근성 인증 레벨 (A, AA, AAA)
□ KRDS 가이드 준수 증빙
□ HTML/CSS/JS 표준 문서와 일치
□ 유지보수 가능한 코드 구조

AI 생성 코드:
❌ 출처 불명확
❌ 접근성 검증 없음
❌ 표준 준수 증빙 불가
❌ 법적 책임 문제

→ 공공기관에서 사용 불가
```

**HANUI의 가치:**

```
"HANUI 기반 프로젝트 = 접근성 심사 통과율 높음"

실무 시나리오:
1. 개발자가 HANUI로 개발
2. 자동 검증 도구로 사전 체크
3. 접근성 인증 심사 → 통과율 95%+
4. 입찰 경쟁력 확보

→ SI 업체에게는 "입찰 무기"
→ 발주처에게는 "리스크 감소"
```

**결론:**

> AI가 아무리 발전해도 **"검증된 결과물"**의 가치는 불변

#### ③ "한국형 UI의 감성과 패턴"

**글로벌 vs 로컬 UI:**

```
Material UI (Google):
- 미국/글로벌 기준
- 기업/스타트업 감성
- 영어 중심 타이포그래피

Ant Design (Alibaba):
- 중국 기준
- 전자상거래 감성
- 중국어 타이포그래피

HANUI:
- 한국 공공기관 기준 ⭐
- 업무용/교육용 감성 ⭐
- 한글 타이포그래피 최적화 ⭐
```

**구체적 차이:**

```
예시: 버튼 디자인

글로벌 기준:
- 버튼 높이: 32px (작고 세련됨)
- 여백: 8px (미니멀)
- 강조색: 원색 계열

한국 공공 기준:
- 버튼 높이: 40px+ (가독성 우선)
- 여백: 12px+ (터치 영역 확보)
- 강조색: 파스텔 톤 (눈의 피로 감소)

→ 한국 사용자 (특히 공무원, 교사, 중장년층)에게
   글로벌 UI는 "불편하고 작다"는 피드백
```

**비유:**

```
미국: Material UI
일본: Nippon Design System
중국: Ant Design
한국: HANUI ⭐
```

**결론:**

> 로컬 감성은 AI가 학습할 수 없는 암묵지(tacit knowledge)

#### ④ "AI와 협업할 수 있는 데이터셋으로서의 가치"

**AI의 한계:**

```
AI가 잘하는 것:
✓ 패턴 기반 코드 생성
  (많이 본 패턴 재생산)

AI가 못하는 것:
❌ 규범 기반 코드 생성
  (명확한 표준 준수)
```

**HANUI의 전략:**

```
KRDS 문서 → HANUI 변환 → AI 학습 가능 형태

1. 정형화된 컴포넌트 코드
   - 재사용 가능한 모듈

2. 메타데이터 (JSON)
   {
     "component": "Button",
     "krds_version": "1.2.0",
     "accessibility": {
       "level": "AA",
       "contrast_ratio": "4.5:1",
       "aria_required": true
     },
     "usage_context": "공공기관 양식 제출",
     "korean_ux_pattern": "확인/취소 버튼 순서"
   }

3. 설명 문서 (한글)
   - "이 버튼은 명도 대비 4.5:1 기준 준수"
   - "공공기관 접근성 심사 통과 기준"
```

**미래 시나리오:**

```
사용자: "HANUI 기준으로 로그인 페이지 만들어줘"

AI (HANUI 학습 후):
✓ KRDS 컬러 토큰 사용
✓ 접근성 마크업 자동 추가
✓ 한국 공공기관 UI 패턴 적용
✓ 행안부 가이드 준수
✓ 검증 가능한 코드

→ 바로 실무 사용 가능
```

**결론:**

> AI 시대엔 HANUI가 오히려 **"AI의 학습 기준서"**가 됨

### 전략 ① "KRDS + 접근성 자동 검수 도구 연계" ⭐ 킬러 피처 (AI 대체 불가)

#### 현재 문제:

```
공공 프로젝트 필수: 한국형 웹 접근성 인증

과정:
1. 개발 완료
2. 접근성 전문가에게 검수 의뢰
3. 지적사항 받음 (20~50개)
4. 개발자가 수정
5. 재검수
6. 반복...

→ 2~4주 소요
→ 비용 500만원~1000만원
→ 일정 지연
```

#### KRDS Platform 해결책:

**A. VSCode Extension "KRDS Validator"**

```typescript
// 개발 중 실시간 검사

<Button>클릭</Button>
     ↑
     ⚠️ Warning: aria-label이 없습니다

💡 Quick Fix:
<Button aria-label="클릭 버튼">클릭</Button>
```

**실시간 검사 항목:**

```
✓ ARIA 속성 (role, label, describedby...)
✓ 키보드 접근성 (tabIndex, onKeyPress...)
✓ 색상 대비 (WCAG AA 이상)
✓ 초점 표시 (focus-visible)
✓ 대체 텍스트 (alt, title)
✓ 폼 레이블 (label for input)
✓ 헤딩 구조 (h1, h2...)
```

**자동 수정 기능:**

```
Ctrl+Shift+P
→ "KRDS: Fix All Accessibility Issues"

변경 사항:
✓ aria-label 자동 추가
✓ alt 속성 생성 (AI)
✓ role 속성 추가
✓ tabIndex 최적화

→ 3분 안에 모든 접근성 이슈 해결
```

**B. CI/CD 통합**

```yaml
# GitHub Actions
name: Accessibility Check

on: [pull_request]

jobs:
  a11y:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: KRDS A11y Check
        run: npx @krds/cli check
      - name: Comment PR
        if: failure()
        run: |
          # PR에 접근성 위반 사항 자동 코멘트
```

**결과:**

```
PR 올리면 자동 검사
→ 접근성 통과 못하면 머지 블락
→ 개발자는 배포 전에 모든 이슈 해결

최종 검수 시:
→ 지적사항 0개
→ 검수 기간 1일
→ 비용 50만원 (형식적 검증만)

시간 절약: 2~4주 → 1일
비용 절약: 500만원 → 50만원
```

#### 경쟁 우위:

```
기존 도구 (Lighthouse, axe-core):
- 범용적 검사 (글로벌 기준)
- 한국 웹 접근성 기준 미흡
- 개발자가 해석/수정

AI 코드 생성 (ChatGPT, v0.dev):
- 접근성 마크업 생성은 함
- 하지만 한국 기준 미준수
- 검증 없음 (법적 책임 없음)

KRDS Validator:
- 한국형 웹 접근성 특화 ⭐
- 자동 수정 기능 ⭐
- KRDS 가이드 연동 ⭐
- 실시간 피드백 ⭐
- 공공기관 검수 통과 보장 ⭐
```

**AI 시대에도 필요한 이유:**

```
AI가 코드를 생성하더라도:
1. 검증이 필요 (HANUI Validator)
2. 한국 기준 준수 확인 필요
3. 법적 책임을 지는 "인증" 필요

→ "AI 생성 + HANUI 검증" = 최적의 조합
```

### 전략 ② "실무형 KRDS Starter Kit"

#### 기존 방식:

```
1. 개발자가 Vite/Webpack 세팅
2. React/Vue 설치
3. KRDS 라이브러리 찾기
4. npm install @krds/something
5. 컴포넌트 하나씩 import
6. 페이지 직접 구성
7. Storybook 세팅 (선택)
8. ESLint 설정
9. 접근성 검사 도구 추가
10. ...

→ 총 1~2주 소요
```

#### HANUI Platform 방식:

```bash
npx create-hanui-app my-project

? Framework? › React
? Include templates? › Yes

✓ 프로젝트 생성 완료! (30초)

cd my-project
npm run dev

→ 이미 작동하는 로그인/대시보드/게시판 포함
→ Storybook, ESLint, 접근성 검사 모두 설정 완료
```

#### 핵심 차별점:

```
단순 컴포넌트 라이브러리 ❌
프로젝트 템플릿 세트 ✅

포함 내용:
├── KRDS 컴포넌트 (커스터마이징 가능)
├── 페이지 템플릿 3개+
├── Storybook 설정
├── ESLint (KRDS 규칙)
├── 접근성 검사 설정
├── 타입스크립트
├── Git 초기화
└── README (프로젝트별 가이드)

→ "바로 개발 시작 가능"
```

#### 벤치마크:

```
Create React App: 기본 세팅만
Next.js: SSR 특화
HANUI Platform: 공공 프로젝트 특화 ⭐
```

**AI 시대 의미:**

```
AI가 코드를 생성해도:
- 프로젝트 구조는 필요
- 설정 파일들은 필요
- 검증된 템플릿은 필요

Starter Kit = AI가 작업할 "캔버스" 제공
```

### 전략 ④ "AI 협업 기능" (보조 도구로 재정의)

> **중요한 전략 변경**
>
> 기존: AI Generator가 "혁신적 킬러 피처"
>
> 변경: AI는 "보조 도구", 핵심은 "검증 + 표준화"

#### 새로운 비전:

```
"AI가 만들고, HANUI가 검증한다"

AI 역할: 코드 생성 보조
HANUI 역할: 표준 준수 + 검증 + 인증

개발 시간:
AI 단독 사용: 30초 생성, 2주 수정 (기준 미준수)
AI + HANUI: 30초 생성, 10분 검증, 바로 사용
```

#### 실제 워크플로우:

**시나리오: 공지사항 게시판 개발**

```
Step 1: AI에게 요청 (ChatGPT, Cursor 등)
───────────────────────────────────────
"공지사항 게시판 만들어줘"

↓ AI가 코드 생성 (30초)

Step 2: HANUI 프로젝트에 복사
───────────────────────────────────────
생성된 코드를 HANUI 프로젝트에 붙여넣기

↓

Step 3: HANUI Validator가 자동 검증
───────────────────────────────────────
❌ KRDS 컬러 토큰 미사용
❌ aria-label 누락 3곳
❌ 명도 대비 부족 (버튼)
⚠️  tabIndex 순서 비추천

↓ Quick Fix 클릭

Step 4: 자동 수정 적용 (10분)
───────────────────────────────────────
✅ KRDS 토큰으로 변환
✅ aria-label 자동 추가
✅ 색상 대비 조정
✅ tabIndex 최적화

↓

Step 5: 최종 검증 및 배포
───────────────────────────────────────
✅ KRDS 준수 100%
✅ 접근성 AA 등급
✅ 공공기관 검수 통과 가능

[Deploy]
```

**핵심:**

```
AI + HANUI = 최고의 조합

AI:
- 빠른 초안 생성
- 반복 작업 자동화

HANUI:
- 한국 기준 준수
- 법적 효력 있는 검증
- 실무 사용 가능한 품질
```

#### 추가 AI 협업 기능 (선택적 구현):

**옵션 1: HANUI Assistant (VSCode 플러그인)**

```
// AI가 HANUI 컴포넌트 추천
개발자: "게시판 목록 만들려고"
Assistant: HANUI Table 컴포넌트 추천
          [Insert Code] 클릭하면 자동 삽입
```

**옵션 2: Figma → HANUI 변환**

```
Figma 플러그인:
1. Figma 디자인 선택
2. "Export to HANUI" 클릭
3. HANUI 컴포넌트 코드 생성
4. KRDS 준수 자동 검증
```

**옵션 3: 기존 사이트 마이그레이션 도구**

```
URL 입력 → 스크린샷 분석 → HANUI 코드 생성
→ 레거시 공공 사이트 현대화
```

#### 중요: AI 기능의 포지셔닝

```
❌ 잘못된 포지셔닝:
"HANUI는 AI로 코드를 생성합니다"
→ ChatGPT와 차별화 안됨

✅ 올바른 포지셔닝:
"HANUI는 AI 생성 코드를 검증하고 표준화합니다"
→ AI 시대에 필수적인 도구

핵심 메시지:
"AI가 생산성을 높이고, HANUI가 품질을 보장합니다"
```

```
→ 공공 프로젝트 필수 도구화
```

#### Phase 3 (5-8개월): Marketplace

```
목표: "KRDS는 여기서 다 구하네"

핵심 차별화:
✅ 커뮤니티 기여 시스템
✅ 수익 분배
✅ KRDS 인증

→ 생태계 구축
→ 자율 성장 시작
```

#### Phase 4 (9-12개월): AI Generator

```
목표: "미래다..."

핵심 차별화:
```
