## 1. 프로젝트 개요

### 1.1 프로젝트명

**KRDS Platform** (가칭: KRDS Kit, KRDS.dev 등)

### 1.2 한 줄 요약

```
"npx create-krds-app"으로 시작하는 공공 웹사이트 개발의 모든 것
- Starter Kit + 컴포넌트 라이브러리 + 마켓플레이스 + AI Generator
```

### 1.3 비전

```
KRDS를 사용하는 모든 개발자의 허브가 되어,
공공 프로젝트 개발 생산성을 10배 향상시킨다
```

### 1.4 미션

- 공공 웹사이트 개발 진입장벽 제거
- KRDS 가이드 준수를 자동화
- 접근성 인증을 쉽게
- 개발자 커뮤니티 활성화

## 2. 문제 정의

### 2.1 현재 상황

KRDS(대한민국 정부 표준 디자인 시스템)는 2023년 공식 발표되었으나:

**문제점들:**

```
❌ HTML 가이드만 존재 (React/Vue 없음)
❌ 개발자가 직접 컴포넌트화 해야 함
❌ 접근성 검증을 매번 수동으로
❌ 프로젝트 세팅에 1~2주 소요
❌ 디자인 가이드 위반 체크 어려움
❌ 재사용 가능한 템플릿 부족
```

### 2.2 기존 시도들의 한계

#### KRDS-community/krds-react

- ✅ React 컴포넌트 제공
- ❌ 문서화 부족 (기술 문서 수준)
- ❌ 프로젝트 템플릿 없음
- ❌ 접근성 자동 검사 없음
- ❌ Star 50개 수준 (인지도 낮음)

#### @krds.ui/vue

- ✅ Vue 3 지원
- ❌ "공식"이라 주장하나 실제 아님
- ❌ 유지보수 의문 (30일 전 업데이트)
- ❌ 생태계 부재

#### @gracefullight/krds (MUI 기반)

- ✅ Material-UI 기반으로 완성도 높음
- ❌ MUI 의존성으로 무거움
- ❌ 커스터마이징 제한적

**공통 문제:**

> 모두 "컴포넌트 라이브러리"일 뿐, **"개발 플랫폼"은 없음**

## 3. 솔루션

### 3.1 핵심 가치 제안

```
단순 라이브러리가 아닌 "KRDS 생태계 플랫폼"

컴포넌트 + 도구 + 템플릿 + 커뮤니티 + AI
= 공공 프로젝트 개발의 모든 것
```

### 3.2 사용자 여정 (User Journey)

#### Before (현재):

```
Day 1-3:   프로젝트 세팅, Webpack/Vite 설정, KRDS 가이드 분석
Day 4-10:  컴포넌트 하나씩 구현 (Button, Input, Table...)
Day 11-15: 접근성 검증, 디자인 QA
Day 16-20: 페이지 구현 시작
...
총 4주 후 첫 페이지 완성
```

#### After (KRDS Platform):

```
Day 1:     npx create-krds-app → 프로젝트 완성
           템플릿 선택 → 로그인/대시보드 자동 생성

Day 2:     필요한 컴포넌트 Marketplace에서 다운로드
           AI Generator로 커스텀 페이지 생성

Day 3:     VSCode Extension이 자동으로 접근성 체크
           자동 수정 기능으로 즉시 해결

Day 4-5:   비즈니스 로직 구현 (본질에 집중)

총 1주 후 프로젝트 완성 (4배 빠름)
```

## 4. 타깃 고객

#### Persona 1: "공공 SI 프론트엔드 개발자"

```
이름: 김개발 (28세)
경력: 3년차 React 개발자

니즈:
- 빠르게 프로젝트 시작하고 싶음
- 접근성은 자동으로 해결되길
- 검증된 템플릿 필요
```

```
이름: 박공무 (35세)
직급: 7급 공무원
기관: 지자체 정보화 담당
페인 포인트:
- SI 업체에 맡기면 결과물 품질 편차 큼
- KRDS 준수 여부 확인 어려움
- 접근성 인증 받는데 시간/비용 많이 듦

니즈:
- 표준화된 결과물
- 검증 가능한 품질
- 빠른 개발/낮은 비용
```

#### Persona 3: "프리랜서 개발자"

```
이름: 이프리 (30세)
경력: 5년차, 1년 전 독립
전문: React, Next.js
페인 포인트:
- 공공 프로젝트 따기 어려움 (KRDS 경험 부족)
- 단가 협상에서 불리 (세팅 시간 포함)
- 혼자라 품질 검증 힘듦

니즈:
- 빠른 납품으로 경쟁력 확보
- 포트폴리오에 쓸 고품질 결과물
- 커뮤니티 지원
```

### 4.2 Secondary Users (간접 사용자)

- **디자이너**: KRDS 준수 디자인 → 개발자에게 전달
- **QA 담당자**: 접근성 자동 검사 도구 활용
- **프로젝트 매니저**: 템플릿으로 견적 산정

### 4.3 시장 규모

```
공공 SI 시장:
- 연간 발주액: 약 3조원
- 웹/앱 프로젝트: 약 30% (9,000억원)
- 프론트엔드 개발자: 추정 5,000명+

잠재 고객:
- SI 업체: 500개사
- 프리랜서: 1,000명
- 공공기관 내부 개발팀: 200개 기관

→ 충분히 큰 시장
```

#### CI/CD 통합

```yaml
# .github/workflows/krds-check.yml
name: KRDS Accessibility Check

on: [pull_request]

jobs:
  a11y-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: KRDS Check
        run: npx @krds/cli check
      - name: Upload Report
        uses: actions/upload-artifact@v3
        with:
          name: a11y-report
          path: ./a11y-report.html
```

### 10.4 AI 시대 리스크 대응 전략 (신규)

#### 리스크: "AI가 HANUI를 대체하면?"

**시나리오:**

```
2026년, GPT-7이 등장하여:
- "KRDS 준수 게시판 만들어줘" → 완벽한 코드 생성
- 접근성도 완벽
- 한국 기준도 학습함
→ HANUI가 불필요해지는 상황
```

**대응 전략:**

**1. 차별화 포인트 재정의**

```
기존 전략:
① Starter Kit
② 접근성 도구
③ Marketplace
④ AI Generator (핵심으로 봄)

AI 시대 전략:
① 접근성 자동 검증 ⭐ (AI 대체 불가)
② KRDS 공식 인증 시스템 ⭐ (법적 효력)
③ 검증된 Marketplace ⭐ (신뢰)
④ AI는 보조 기능 (주력 아님)

핵심 변화:
"AI 생성"에서 "AI 검증"으로 포지셔닝 전환
```

**2. 근본적 경쟁력 확보**

```
AI가 대체 못하는 영역에 집중:

① 로컬 표준화
   - 한국 공공 디자인 문맥
   - AI 학습 데이터셋 역할

② 검증과 인증
   - 공공기관 납품 필수 요건
   - 법적 책임

③ 한국형 감성
   - 암묵지(tacit knowledge)
   - 사용자 경험 최적화

④ 생태계
   - 커뮤니티는 AI가 못만듦
   - 네트워크 효과
```

**3. AI와의 협업 전략**

```
경쟁 관계 → 협력 관계

사용자 워크플로우:
1. AI (ChatGPT 등)로 초안 생성
2. HANUI 프로젝트에 복사
3. HANUI Validator가 자동 검증
4. Quick Fix로 KRDS 기준 준수
5. HANUI 인증 마크 획득
6. 공공기관 납품

→ "AI + HANUI = 최적의 조합"
```

**4. 글로벌 확장 전략**

```
영문명: HanUI - Korean Design System for Public Web

확장 방향:
- 일본 (유사한 공공 접근성 기준)
- 대만 (정부 디자인 시스템 존재)
- 동남아 (한국 영향력 큰 국가)

차별화:
"AI는 글로벌, HanUI는 로컬 전문가"
```

**5. 측정 지표**

```
AI 시대 대응 성공 지표:

✅ HANUI Validator 사용률 > 프로젝트 생성률
   (검증 도구로서의 가치)

✅ "AI + HANUI" 언급 증가
   (AI와의 협업 인식)

✅ 공공기관 공식 인증 획득
   (대체 불가능한 지위)

✅ AI 플랫폼과의 파트너십
   (v0.dev, Cursor 등과 통합)
```
